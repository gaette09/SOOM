# SOOM Subjective Check-in UX Spec

## A. Purpose

Subjective Check-in은 사용자가 직접 느끼는 몸 상태를 짧게 기록하는 흐름이다. SOOM Recovery v2에서는 운동 기록만으로 알기 어려운 체감 피로, 수면감, 근육통, 기분을 회복 해석의 보완 데이터로 활용한다.

목적:

- 사용자의 주관적 컨디션을 10초 안에 기록한다.
- Recovery score와 coach message가 사용자의 실제 몸 상태에 더 가까워지도록 돕는다.
- 운동 부하, HRV, 수면 같은 정량 데이터가 부족하거나 충돌할 때 해석의 보조 신호로 사용한다.
- 사용자가 평가받는 느낌이 아니라 “오늘 몸 상태를 남겨두는” 경험을 만든다.

## B. UX Principle

Check-in은 가볍고 부담 없어야 한다. SOOM은 사용자의 건강 상태를 판정하지 않고, 운동 컨디션을 이해하기 위한 기록 경험을 제공한다.

원칙:

- 10초 이내 입력을 목표로 한다.
- 하루 1회 입력을 중심으로 설계한다.
- Morning Check-in은 v1의 기본 사용 루프이며, Daily Readiness를 확인한 뒤 오늘 컨디션을 가볍게 남기는 흐름으로 설계한다. 자세한 흐름은 [SOOM_MORNING_CHECKIN_FLOW.md](SOOM_MORNING_CHECKIN_FLOW.md)를 따른다.
- 운동 전/후 입력도 가능하지만 강제하지 않는다.
- v1 저장 정책은 로컬 우선이며, 자세한 persistence 방향은 [SOOM_CHECKIN_PERSISTENCE_PLAN.md](SOOM_CHECKIN_PERSISTENCE_PLAN.md)를 따른다.
- 의료, 진단, 질병 판정처럼 보이는 표현을 피한다.
- 사용자가 “잘했다/못했다”로 평가받는 느낌을 주지 않는다.
- 부드럽고 코칭 중심의 문장을 사용한다.
- 입력하지 않아도 앱의 기본 Recovery 경험은 계속 동작해야 한다.

권장 문장 톤:

- “오늘 몸 상태를 짧게 남겨볼까요?”
- “기록해두면 회복 추천이 조금 더 개인화돼요.”
- “정답은 없어요. 지금 느끼는 상태만 골라주세요.”

피해야 할 문장:

- “피로도가 위험합니다.”
- “수면 상태가 나쁩니다.”
- “운동하면 안 됩니다.”
- “정확한 건강 평가를 위해 반드시 입력하세요.”

## C. Input Fields v1

v1 입력은 한 화면에서 끝나는 5단계 선택형으로 시작한다. 모든 값은 `1...5` 범위를 사용하고, 선택하지 않은 항목은 저장하지 않거나 기본 중립값으로 처리할 수 있다. 단, 계산 계층으로 들어가기 전에는 모델에서 범위를 보정한다.

### fatigueLevel

- 한국어 라벨: 피로감
- 짧은 설명: 지금 몸이 얼마나 피곤하게 느껴지는지 기록한다.
- 1점 의미: 아주 가벼움
- 5점 의미: 매우 피곤함
- 추천 UI: 5단계 선택 버튼
- 예시 선택지: 아주 가벼움 / 가벼움 / 보통 / 피곤함 / 매우 피곤함

### sleepQuality

- 한국어 라벨: 수면감
- 짧은 설명: 지난밤 잠이 얼마나 개운했는지 기록한다.
- 1점 의미: 거의 회복되지 않음
- 5점 의미: 아주 개운함
- 추천 UI: 5단계 선택 버튼
- 예시 선택지: 매우 부족 / 부족 / 보통 / 좋음 / 아주 좋음

### muscleSoreness

- 한국어 라벨: 근육통
- 짧은 설명: 운동에 영향을 줄 정도의 뻐근함이나 통증 느낌을 기록한다.
- 1점 의미: 거의 없음
- 5점 의미: 매우 뻐근함
- 추천 UI: 5단계 선택 버튼
- 예시 선택지: 없음 / 약간 / 보통 / 뻐근함 / 매우 뻐근함

### moodLevel

- 한국어 라벨: 기분
- 짧은 설명: 오늘 훈련을 시작할 때의 마음 상태를 기록한다.
- 1점 의미: 무겁고 의욕이 낮음
- 5점 의미: 가볍고 의욕이 좋음
- 추천 UI: 5단계 선택 버튼
- 예시 선택지: 무거움 / 낮음 / 보통 / 좋음 / 매우 좋음

### note

- 한국어 라벨: 메모
- 짧은 설명: 컨디션에 영향을 준 일을 짧게 남긴다.
- 입력 의미: 선택 입력이며, 비워도 된다.
- 추천 UI: 한 줄 또는 짧은 멀티라인 텍스트 입력
- 예시 placeholder: “예: 늦게 잠, 장거리 운전, 다리 뻐근함”

## D. Suggested UI Pattern

Check-in 화면은 슬라이더보다 5단계 선택 버튼을 우선한다. 슬라이더는 정밀해 보이지만 10초 입력에는 부담이 커질 수 있고, 사용자가 “정확한 수치”를 맞춰야 한다는 느낌을 받을 수 있다.

권장 패턴:

- 한 화면에서 모든 항목을 입력한다.
- 각 항목은 라벨, 짧은 설명, 5단계 선택 버튼으로 구성한다.
- 이모지와 텍스트 조합을 사용할 수 있지만, 과도하게 귀엽거나 게임처럼 보이지 않게 한다.
- 기본 선택값은 중립 또는 미선택 상태를 사용할 수 있다.
- 저장 버튼은 “저장” 또는 “오늘 상태 기록”처럼 단순하게 둔다.
- 저장 후에는 짧은 피드백을 제공한다.

저장 후 피드백 예:

- “오늘의 회복 해석이 더 정확해졌어요.”
- “기록했어요. 다음 추천에 반영할게요.”
- “좋아요. 오늘 컨디션 흐름에 함께 반영할게요.”

입력 후 Recovery 화면:

- 저장된 최신 check-in은 Recovery 화면에서 “오늘 컨디션 기록” 요약 카드로 보여준다.
- 피로감, 수면감, 근육통, 기분, 선택 메모를 짧게 확인할 수 있게 한다.
- 이 요약은 v1 점수 계산에는 반영하지 않고, 사용자가 기록이 저장되었음을 확인하는 피드백 루프로만 사용한다.
- Recovery 화면의 핵심 판단은 회복 점수, 코치 메시지, 추천 행동이 먼저 담당한다. Check-in 요약은 그 판단을 보완하는 보조 맥락으로 배치한다.

기록 확인 화면:

- Check-in History는 사용자가 저장된 컨디션 기록을 확인하는 조회 전용 화면으로 시작한다.
- 목록에는 날짜, 피로감, 수면감, 근육통, 기분, 선택 메모를 짧게 보여준다.
- v1 History MVP에서는 수정/삭제 UI를 제공하지 않는다. 수정/삭제는 persistence 정책과 확인 문구가 안정화된 다음 단계에서 연결한다.
- 기록 확인은 평가나 분석보다 “내가 남긴 몸 상태를 다시 보는” 조용한 경험이어야 한다.

기록 상세 화면:

- Check-in Detail MVP는 개별 기록을 더 자세히 보는 조회 전용 화면이다.
- 날짜, 피로감, 수면감, 근육통, 기분, 전체 메모를 표시한다.
- 메모가 없으면 “메모가 없어요.”처럼 부드러운 빈 상태를 보여준다.
- 상세 화면에는 “이 기록은 회복 코칭 문장을 더 개인화하는 데 사용됩니다.” 수준의 짧은 안내를 둔다.
- Detail MVP 이후 Edit MVP에서는 우측 상단 “수정하기” 진입으로 기존 기록을 수정할 수 있다.
- 수정 후 상세 화면의 값은 즉시 갱신되며, Recovery 화면을 다시 로드하면 최신 기록 기반 coach message/insight 개인화 문구도 업데이트될 수 있다.
- 삭제는 Edit MVP 이후 Delete MVP에서 개별 기록 단위로 제공한다.

수정/삭제 UX 원칙:

- 사용자가 실수로 잘못 입력한 경우 최신 check-in을 부담 없이 수정할 수 있어야 한다.
- v1 Edit MVP는 피로감, 수면감, 근육통, 기분, 메모를 같은 5단계 선택 패턴으로 수정한다.
- 수정은 의료 설문을 고치는 느낌이 아니라 “오늘 기록을 조금 정리하는” 가벼운 흐름이어야 한다.
- 삭제는 사용자가 자기 기록을 통제하는 기본 기능으로 제공한다.
- 단일 기록 삭제는 가볍게 처리하되, 전체 삭제는 짧은 확인 문구를 둔다.
- v1 Delete MVP는 개별 기록 삭제만 제공하며, 전체 삭제 UI는 설정/개인정보 메뉴가 준비될 때까지 보류한다.
- 삭제 확인 문구는 “이 기록을 삭제할까요?”처럼 명확하게 묻고, “삭제해도 회복 점수는 다시 계산되지 않아요.”처럼 v1 정책을 부드럽게 안내한다.
- “건강 데이터 삭제”처럼 과도하게 무겁거나 불안하게 표현하지 않고, “오늘 컨디션 기록 삭제”처럼 명확하고 차분하게 표현한다.
- 삭제 후에는 Recovery score가 갑자기 변했다는 느낌을 주지 않는다. v1.5에서는 check-in이 score/status/recommendation이 아니라 최신 요약과 coach/insight 개인화에만 영향을 준다는 정책을 유지한다.

## E. Frequency & Timing

Check-in은 사용자의 루틴을 방해하지 않아야 한다.

후보 타이밍:

- 아침 체크인: 오늘의 훈련 준비도와 가장 잘 연결된다.
- 운동 직후 체크인: 세션의 체감 강도와 회복 필요도를 보완한다.
- 잠들기 전 체크인: 하루 피로와 수면 전 컨디션을 기록할 수 있다.

v1 권장:

- 하루 1회 아침 체크인을 기본 흐름으로 둔다.
- Morning Check-in은 Daily Readiness를 보조하는 lightweight prompt로 사용하며, 기록하지 않아도 Recovery 화면은 정상 동작한다.
- 운동 직후/잠들기 전 체크인은 v2 이후 확장 후보로 둔다.
- 알림을 사용하더라도 강제감이 없게 한다.
- 연속 기록 실패를 벌점처럼 표현하지 않는다.

## F. Privacy / Trust Copy

Check-in은 민감하게 느껴질 수 있는 몸 상태 데이터다. 사용자가 왜 기록하는지, 기록하지 않아도 되는지, 의료 진단이 아니라는 점을 명확히 알려야 한다.

권장 카피:

- “이 데이터는 회복 추천을 더 개인화하는 데 사용됩니다.”
- “언제든 기록하지 않아도 됩니다.”
- “의료 진단이 아닌 운동 컨디션 참고용입니다.”
- “기록은 회복 흐름을 이해하기 위한 참고 데이터로 사용됩니다.”

피해야 할 카피:

- “정확한 분석을 위해 반드시 입력해야 합니다.”
- “건강 상태를 진단합니다.”
- “입력하지 않으면 회복 점수가 부정확합니다.”

## G. Recovery Formula v2 Integration

Subjective Check-in은 v2에서 Recovery score와 coach message를 보정하는 입력 후보로 사용한다. v1에서는 모델과 store, provider 준비까지만 하고 점수에는 반영하지 않는다.

v1.5 정책:

- check-in은 Recovery score, status, recommendation을 변경하지 않는다.
- 사용자가 입력한 피로감, 수면감, 근육통, 기분은 coach message와 insight 개인화에만 사용한다.
- v1.5 개인화는 `RecoverySummaryComposer`를 통해 activity 기반 summary 위에 적용한다.
- `RecoveryCheckInSignalClassifier`는 피로감, 수면감, 근육통, 기분 우선순위를 중앙화한다.
- insight 개인화는 한 번에 1개만 추가하고, 기존 Recovery 분석 흐름 앞에서 부드러운 보조 맥락으로 보여준다.
- 개인화 문장은 진단이 아니라 “오늘 기록한 컨디션을 참고한 부드러운 코칭”으로 표현한다.
- check-in fetch에 실패해도 기존 activity 기반 Recovery 화면은 그대로 동작해야 한다.

v2 반영 방향:

- `fatigueLevel`이 높으면 recovery score를 소폭 감소시킨다.
- `sleepQuality`가 높으면 recovery score를 소폭 증가시킨다.
- `muscleSoreness`가 높으면 고강도 추천을 제한한다.
- `moodLevel`은 coachMessage tone에 반영한다.
- `note`는 v2에서는 저장만 하고, v3에서 AI summary 후보로 사용한다.

점수 반영 원칙:

- check-in 하나만으로 점수를 크게 흔들지 않는다.
- 운동 부하, 수면, HRV 등 다른 신호와 충돌하면 `dataQuality` 또는 설명 문구로 투명하게 안내한다.
- 낮은 점수는 경고가 아니라 회복 전략으로 표현한다.
- 사용자의 직접 입력은 “판정”보다 “맥락”으로 다룬다.

## H. Empty / Skip State

Check-in은 선택 경험이어야 한다. 입력하지 않아도 Recovery v1은 activity 기반으로 계속 동작한다.

빈 상태 원칙:

- 체크인이 없어도 Recovery 화면은 정상 표시된다.
- 체크인이 없으면 향후 `dataQuality`를 낮출 수 있지만, 사용자를 압박하지 않는다.
- “기록하면 더 좋아져요” 정도의 부드러운 안내만 사용한다.
- skip은 실패가 아니라 자연스러운 상태로 다룬다.
- Morning Check-in의 `나중에` 선택은 오늘 하루만 적용한다. 앱을 재실행해도 오늘은 prompt를 다시 띄우지 않지만, 날짜가 바뀌면 다시 기록을 제안할 수 있다.
- skip은 check-in history에 저장하지 않는다. 사용자의 몸 상태 기록이 아니라 prompt 표시 정책이다.

빈 상태 카피 예:

- “오늘 컨디션 기록은 건너뛰었어요.”
- “기록하지 않아도 회복 추천은 계속 볼 수 있어요.”
- “짧게 기록하면 추천을 조금 더 개인화할 수 있어요.”

## I. Future v2/v3

v2 후보:

- 주간 컨디션 추세
- Recovery score 보정
- 고강도 추천 제한
- Coach message tone 개인화
- 아침 체크인 알림

v3 후보:

- AI 코치 문장 개인화
- HRV, 수면, 운동부하와 check-in 결합
- note 기반 주간 컨디션 요약
- 간단한 음성 입력
- 대회 준비 기간, 생리 주기, 여행/시차 같은 맥락 입력

확장 시에도 Check-in은 짧고 조용해야 한다. SOOM의 핵심은 사용자를 더 바쁘게 만드는 것이 아니라, 복잡한 몸 상태 데이터를 이해하기 쉽게 정리해주는 것이다.
