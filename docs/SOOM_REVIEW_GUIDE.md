# SOOM Review Guide

SOOM은 iOS Native 앱이며, SwiftUI 기반으로 개발한다.

## 제품 방향

- Apple Fitness의 부드럽고 정돈된 감성
- Strava의 운동 데이터 중심 UX
- 복잡한 운동 데이터를 초보자도 이해하기 쉽게 보여주는 앱
- 브랜드 느낌은 차분하고 신뢰감 있으며, 과도하게 귀엽거나 게임스럽지 않게 한다
- 장기적으로 회복, 피로도, 운동 습관, AI 코칭까지 확장할 수 있는 구조를 유지한다
- HealthKit 연결 전에는 화면이 필요로 하는 데이터 계약을 먼저 정의한다

## 리뷰 원칙

SOOM의 리뷰는 단순히 화면이 예쁜지 보는 것이 아니라, 사용자가 운동 데이터를 더 잘 이해하고 다음 행동을 결정할 수 있는지 확인하는 과정이다. 모든 리뷰는 제품 목적, iOS 네이티브 품질, 디자인 시스템 일관성, 장기 확장성을 함께 본다.

리뷰 에이전트는 아래 문서를 함께 기준으로 삼는다.

- `docs/SOOM_DESIGN_SYSTEM.md`
- `docs/SOOM_ICON_SYSTEM.md`

## 1. Product Fit

### 체크리스트

- 화면이 SOOM의 핵심 가치인 운동 분석, 회복 판단, 다음 행동 제안에 기여하는가?
- 운동 초보자도 데이터의 의미를 바로 이해할 수 있는가?
- 고급 사용자가 기대하는 페이스, 심박, 파워, 스플릿, 지도 경로 정보가 충분히 드러나는가?
- 커뮤니티/SNS 기능이 운동 데이터 중심 경험을 해치지 않는가?
- 브랜드 톤이 차분하고 신뢰감 있게 유지되는가?

### 좋은 예

- "최근 30일 러닝 볼륨이 29% 증가했으니 다음 주는 회복 러닝을 섞으세요"처럼 데이터와 행동 제안을 연결한다.
- 그래프와 요약 카피가 함께 있어 숫자를 몰라도 의미를 이해할 수 있다.
- 피드 게시물에서도 운동 핵심 지표가 먼저 보이고, 반응/댓글은 보조 역할을 한다.

### 나쁜 예

- 운동 기록보다 장식 이미지나 감성 문구가 먼저 보인다.
- "좋아요", "팔로워", "배지" 같은 소셜 요소가 운동 분석보다 강하게 노출된다.
- 데이터는 많지만 사용자가 무엇을 해야 할지 알 수 없다.

## 2. UX Flow

### 체크리스트

- 홈에서 상태 파악, 분석 확인, 운동 상세 진입까지 흐름이 자연스러운가?
- 상세 페이지에서 지도, 요약, 그래프, 스플릿, AI 해석 순서가 논리적인가?
- 사용자가 현재 위치한 화면과 다음 행동을 쉽게 이해할 수 있는가?
- 시트, 탭, 뒤로가기, 저장, 공유 등 주요 조작이 iOS 사용 습관과 맞는가?
- 스크롤, 드래그, 전환 애니메이션이 예측 가능하고 끊기지 않는가?

### 좋은 예

- 운동 상세 첫 진입 시 지도 경로와 핵심 요약을 동시에 보여준다.
- 상세 레이어는 중앙, 상단, 하단 상태가 명확하고 각 상태에서 가능한 제스처가 일관된다.
- 상단 고정 상태에서만 내부 스크롤이 가능해 사용자가 레이어 이동과 콘텐츠 스크롤을 혼동하지 않는다.

### 나쁜 예

- 같은 드래그 동작이 어떤 때는 내부 스크롤, 어떤 때는 레이어 이동으로 불규칙하게 반응한다.
- 상세 화면에서 마지막 콘텐츠가 하단에 잘려 보인다.
- 뒤로가기, 닫기, 저장 버튼 위치가 화면마다 다르게 배치된다.

## 3. UI Consistency

### 체크리스트

- 색상, 폰트, 카드, 버튼, 탭바, 아이콘 크기가 디자인 시스템을 따르는가?
- 같은 정보 유형은 같은 컴포넌트와 시각 계층으로 표현되는가?
- 상세 페이지, 피드 상세, 홈 카드가 서로 다른 앱처럼 보이지 않는가?
- 여백, 라운드, 라인, 그림자 강도가 과하지 않고 일관되는가?
- 운동 종목 색상과 상태 색상이 혼동되지 않는가?

### 좋은 예

- 수영은 `#0047AB`, 사이클은 `#16302B`, 러닝은 `#B80F0A`, 경고/피로는 `#FF8F00`를 일관되게 사용한다.
- 큰 제목은 Gmarket Sans, 본문은 NanumSquare Neo를 사용한다.
- 카드 반경과 내부 여백이 화면 전반에서 일정하다.

### 나쁜 예

- 같은 러닝 데이터가 어떤 화면에서는 빨간색, 다른 화면에서는 주황색으로 표시된다.
- 버튼마다 아이콘 크기, 터치 영역, 배경 스타일이 다르다.
- 한 화면 안에서 카드 스타일과 섹션 스타일이 섞여 시각적 우선순위가 흐려진다.

## 4. iOS Native Quality

SOOM v1은 브랜드 일관성과 초기 UI 완성도를 위해 Light Mode only 정책을 사용한다. Dark Mode는 디자인 시스템 안정화 이후 v2에서 지원한다.

### 체크리스트

- SwiftUI 네이티브 컴포넌트와 iOS 제스처 관성이 자연스럽게 느껴지는가?
- Safe Area, Dynamic Island, 홈 인디케이터, 상태바 영역을 올바르게 처리하는가?
- 실제 기기와 시뮬레이터에서 레이아웃이 동일하게 안정적인가?
- MapKit, Charts, ScrollView, NavigationStack 사용이 iOS 방식에 맞는가?
- Light Mode only 정책과 코드/문서가 충돌하지 않는가?
- 텍스트 크기 변화에 취약하지 않은가?

### 좋은 예

- 지도 상세 화면에서 상태바와 상단 헤더 배경이 하나의 레이어처럼 보인다.
- 드래그와 스크롤 전환이 Apple 앱처럼 부드럽고 예측 가능하다.
- MapKit 경로가 상세 레이어 상태에 맞춰 중앙 정렬과 줌 레벨을 자연스럽게 조정한다.

### 나쁜 예

- 상태바 위로 콘텐츠가 겹치거나 배경색이 분리되어 보인다.
- 하단 홈 인디케이터 영역과 탭바, 콘텐츠가 서로 겹친다.
- SwiftUI 상태 변화가 과도한 애니메이션이나 튐 현상을 만든다.

## 5. Design System

### 체크리스트

- 색상 토큰, 폰트 토큰, 레이아웃 토큰을 직접 값 대신 사용했는가?
- 새로운 UI가 기존 `SOOMColor`, `SOOMFont`, `SOOMLayout` 규칙을 따른다.
- 컴포넌트가 반복 가능한 형태로 분리되어 있는가?
- Metric UI가 `SOOMMetricPill`, `SOOMMetricRing`, `SOOMMetricRow`, `SOOMActionRow`처럼 표시 책임별로 나뉘어 있는가?
- `Components`에 들어간 View가 특정 Feature의 ViewModel이나 모델 흐름에 의존하지 않는가?
- 디자인 가이드 문서와 실제 구현이 어긋나지 않는가?
- 새 패턴이 필요한 경우 문서에 먼저 정의했는가?

### 좋은 예

- `SOOMCard`, `SOOMSectionHeader`, Metric 계열 컴포넌트 같은 재사용 컴포넌트를 활용한다.
- 색상은 `SOOMColor.run`, `SOOMColor.bike`, `SOOMColor.swim`처럼 의미 기반으로 사용한다.
- 상세 레이어의 높이, 여백, 스냅 위치는 한 곳에서 관리된다.

### 나쁜 예

- 화면마다 `Color(hex:)`, `.font(.system...)`, 임의 padding 값을 직접 흩뿌린다.
- 기존 카드 컴포넌트가 있는데 비슷한 카드를 새로 만든다.
- 디자인 가이드와 실제 앱 색상/폰트가 다르게 적용된다.
- 피드/커뮤니티 화면이 Activity Feature 안에 계속 쌓인다.

## 6. Code Architecture

### 체크리스트

- View, ViewModel, Mock/Harness 데이터 역할이 분리되어 있는가?
- 화면 상태와 데이터 상태가 명확히 나뉘어 있는가?
- HealthKit 또는 외부 API를 붙이기 전에 화면 데이터 계약이 문서와 모델로 먼저 정의되어 있는가?
- 상세 페이지, 피드 상세, 운동 상세가 중복 없이 공유 구조를 사용하는가?
- 새로운 기능이 기존 구조를 깨지 않고 확장 가능한가?
- 더미 데이터에서 실제 데이터로 전환하기 쉬운가?

### 좋은 예

- 운동 데이터는 `MockWorkoutHarness` 같은 데이터 공급 계층을 통해 주입된다.
- 지도 상세 레이아웃은 공통 scaffold로 만들고, 내부 콘텐츠만 운동 상세/피드 상세에 맞게 바꾼다.
- 계산 로직은 View 안에 과도하게 쌓지 않고 모델 또는 헬퍼로 분리한다.

### 나쁜 예

- 하나의 View 파일에 화면, 데이터 생성, 비즈니스 로직, 포맷팅이 모두 섞인다.
- 피드 상세와 운동 상세가 거의 같은 코드를 복사해서 따로 유지한다.
- 실제 HealthKit, Strava API, 서버 데이터로 전환할 때 대규모 재작성해야 한다.

## 7. Accessibility

### 체크리스트

- VoiceOver에서 버튼과 주요 데이터가 의미 있게 읽히는가?
- 색상만으로 상태를 구분하지 않는가?
- Dynamic Type 확장 시 텍스트가 잘리거나 겹치지 않는가?
- 터치 영역이 iOS 권장 크기를 만족하는가?
- 그래프와 지도 정보에 대체 설명 또는 요약 정보가 있는가?

### 좋은 예

- "러닝, 10.4km, 52분, 평균 심박 151bpm"처럼 운동 카드가 하나의 의미 있는 문장으로 읽힌다.
- 위험, 피로, 회복 상태는 색상뿐 아니라 텍스트 라벨과 수치로 함께 표시한다.
- 작은 아이콘 버튼에도 접근성 라벨이 있어 기능을 알 수 있다.

### 나쁜 예

- 아이콘만 있고 VoiceOver에서 "button"으로만 읽힌다.
- 빨간색/초록색만으로 좋음과 위험을 구분한다.
- 글자 크기를 키우면 카드 안의 숫자와 라벨이 겹친다.

## 8. Scalability

### 체크리스트

- 운동 종목, 지표, 그래프, 분석 카드가 추가되어도 구조가 유지되는가?
- 실제 HealthKit, Strava, 서버 API, AI 분석 결과를 연결하기 쉬운가?
- 커뮤니티, 클럽, 피드, 알림 기능 확장에 대비되어 있는가?
- 데이터 양이 늘어나도 화면 성능과 스크롤 성능이 유지되는가?
- 새로운 디자인 패턴을 추가할 때 기존 시스템과 충돌하지 않는가?

### 좋은 예

- 러닝, 사이클, 수영 외 종목이 추가되어도 동일한 상세 컴포넌트 구조를 재사용할 수 있다.
- 그래프 데이터가 늘어나도 Charts 컴포넌트가 명확한 입력 모델을 받는다.
- AI 분석 결과는 카드 단위로 확장되어 여러 종류의 코칭 메시지를 담을 수 있다.

### 나쁜 예

- 특정 종목 이름이나 지표가 View 내부에 하드코딩되어 있다.
- 더미 데이터 구조와 실제 API 데이터 구조가 완전히 달라 연결 시 화면을 다시 만들어야 한다.
- 기능이 늘어날수록 View 파일이 계속 커져 유지보수가 어려워진다.
